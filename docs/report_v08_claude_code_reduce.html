CS5.7 Integridad de los artefactos generados e incorporados al SaMD B/C Cumple parcialmente Anthropic documenta parcialmente el proceso de generación de salidas y ofrece mecanismos de trazabilidad, pero con limitaciones relevantes para el contexto SaMD. A favor: (1) La API devuelve en cada respuesta el campo 'model' con el identificador exacto del modelo que generó la salida (p. ej. 'claude-sonnet-4-20250514'), junto con un 'id' de mensaje único y cabeceras 'request-id'. (2) Claude Code emite trazabilidad vía OpenTelemetry: el span 'claude_code.llm_request' registra model name, latencia y token counts por cada llamada; el atributo 'prompt.id' vincula todos los eventos de un prompt. (3) La propagación W3C TRACEPARENT conecta spans del cliente con la traza servidor. Sin embargo, existen brechas: (a) la trazabilidad OTel es opt-in y no está activa por defecto; (b) el texto de respuesta del modelo ('assistant response text') no se recoge salvo activación explícita; (c) no existe un mecanismo nativo de firma o hash de artefactos generados para garantizar su integridad inmutable; (d) los logs de auditoría del plan Pro/Max/Team no incluyen el modelo invocado ni el contenido de las conversaciones. CS5.11 Mantenibilidad y soporte activo del modelo o versión utilizada C/O Cumple parcialmente Anthropic dispone de una política de ciclo de vida de modelos públicamente documentada con cuatro estados (Active → Legacy → Deprecated → Retired) y garantiza al menos 60 días de preaviso antes del retiro de modelos públicamente lanzados, notificando por email y en la documentación oficial. Los modelos activos en el momento de la evaluación (familia Claude 4.x/Sonnet 5) tienen soporte activo. Sin embargo, existen dos limitaciones relevantes para SaMD: (1) el preaviso mínimo garantizado de 60 días puede ser insuficiente para los plazos regulatorios de validación y revalidación de software médico bajo IEC 62304; (2) los modelos que Claude Code utiliza internamente no son configurables por el usuario en el plan Pro/Max/Team, lo que impide anclar una versión concreta del modelo subyacente, exponiendo al fabricante a cambios no controlados. La organización puede mitigar esto mediante procedimientos de monitorización de la página de deprecaciones y planes de actualización documentados, pero la falta de control sobre la versión exacta del modelo en el CLI representa una brecha real de mantenibilidad. https://docs.anthropic.com/en/docs/about-claude/model-deprecations (política oficial de ciclo de vida y preaviso de 60 días) Establecer un proceso de seguimiento del ciclo de vida del modelo utilizado.
CS5.12 Controles de seguridad del modelo B/C Cumple parcialmente Anthropic implementa controles documentados contra prompt injection en Claude Code a múltiples capas: (1) sistema de permisos read-only por defecto con aprobación explícita para operaciones sensibles; (2) sandboxing con aislamiento de filesystem y red basado en primitivas OS (bubblewrap/macOS Seatbelt), que contiene el impacto de inyecciones exitosas; (3) clasificadores de entrada que escanean contenido no confiable en el contexto del modelo; (4) clasificador de salida (auto mode) que valida acciones contra la intención del usuario antes de ejecutarlas; (5) entrenamiento con reinforcement learning para resistencia nativa a inyecciones. Sin embargo, existen evidencias públicas de vulnerabilidades CVE graves patched y activas: CVE-2025-54794 (CVSS 7.7), CVE-2025-54795 (CVSS 8.7), CVE-2025-59536 (CVSS 8.7), CVE-2026-21852, y nuevas PoC publicadas en junio 2026. El propio system card de Anthropic reconoce que el Security Review de Claude Code 'no está endurecido contra prompt injection'. El sandboxing es opt-in y no activo por defecto. Los controles del proveedor son sustanciales pero la brecha residual es real y documentada. https://code.claude.com/docs/en/security | https://www.anthropic.com/engineering/claude-code-sandboxing | https://www.anthropic.com/engineering/claude-code-auto-mode | https://www.anthropic.com/research/prompt-injection-defenses | https://www.anthropic.com/transparency | CVE-2025-54794, CVE-2025-54795 (Cymulate/cymulate.com) | CVE-2025-59536, CVE-2026-21852 (thehackernews.com/2026/02) | https://www.anthropic.com/engineering/how-we-contain-claude Evaluar si los controles del proveedor son suficientes para el caso de uso previsto.
CS5.13 Cifrado y aislamiento de datos entre clientes B Cumple Anthropic documenta públicamente cifrado en tránsito (TLS 1.2+) y en reposo (AES-256) para todos los datos de Claude Code, confirmado tanto en la documentación oficial de Claude Code (code.claude.com/docs/en/data-usage) como en el Anthropic Privacy Center. La telemetría (vía Sentry) también usa cifrado en tránsito TLS y en reposo AES-256.
